# Data Model — Scaffold & Design System (Mercado Pop)

**Feature**: 007 | **Plan**: [plan.md](./plan.md) | **Date**: 2026-06-17

This feature ships **no persistent entities** — there is no DB schema change, no migration, no Drizzle table. What follows is the set of **design-time contracts** the feature codifies. Treat each section as the canonical shape that downstream features must read from and conform to.

---

## 1. Design Token Set

The Mercado Pop token surface from `docs/business/overview.md` §2. Every token below is exposed in BOTH `app/assets/css/tokens.css` (as a CSS custom property) AND `tailwind.config.ts` (as a Tailwind theme value). The two representations MUST resolve to identical computed values.

### Palette override (token enforcement)

`tailwind.config.ts` **overrides** `theme.colors` (it does NOT use `theme.extend.colors`). The consequence is binding: Tailwind's default palette (`slate`, `gray`, `red`, `orange-500`, `blue-700`, etc.) is NOT generated by the build. The only non-token color names preserved are `transparent` and `currentColor`. A typo or a "let me just grab `bg-orange-500`" produces NO compiled class — the offending surface drifts visually, which is the intended early-warning signal. Adding a new color (neutral, status, decorative) REQUIRES adding it first to `tokens.css` and to the override map in `tailwind.config.ts` — there is no fallback. Enforcement is mechanical (see `tasks.md` T004 + T108a/b/c).

## 2. Accent Scope

A single CSS class that swaps the local `--accent` from orange to blue. Every component reads `--accent` (directly or via `bg-accent`/`text-accent`/`border-accent`); wrapping a region in `.scope-express` re-points the variable.

```css
.scope-express { --accent: var(--blue); }
```

**Contract**:
- `--orange` (`#FF6B2B`) and `--blue` (`#2E7CF6`) are **peer brand tokens**. Their token names MUST surface this peerage (`--orange` / `--blue`, `bg-orange` / `bg-blue`, and any namespaced brand surface like `brand.ayce` / `brand.express`). Names that subordinate Express ("secondary", "accent-2", etc.) are PROHIBITED.
- Express styling MUST be triggered ONLY by wrapping a subtree in `.scope-express` (or by `Card`/`Chip`/`Nav` accepting `accent="express"`, which sets the class on the rendered root).
- No component MAY hard-code `var(--blue)` or `bg-blue` in its **accent path** — it MUST read through `--accent`. (The peer-token rule applies to token names and exposure surface, not to the per-component code path.)
- Grep enforcement: the codebase MUST contain exactly ONE declaration of `--accent: var(--blue)` (the scope class itself).
- Downstream features 010 (Menu — Express tab) and 012 (Branches — SUMO Express markers) toggle this scope based on their state.

## 3. i18n Locale File

Two JSON files under `i18n/locales/` — one per locale. Structure is keyed by `section.subsection.key`. Missing keys fall back to the key name (configured via the i18n module).

### Seed key set (this feature)

```json
{
  "nav": {
    "menu": "...",
    "promociones": "...",
    "sucursales": "...",
    "lealtad": "...",
    "contacto": "...",
    "reservar": "..."
  },
  "common": {
    "cta": {
      "reserve": "...",
      "viewMenu": "...",
      "findBranch": "..."
    },
    "lang": {
      "toggle": "..."
    }
  },
  "brand": {
    "tagline": "...",
    "ayceBadge": "..."
  }
}
```

### Locked values (per `docs/business/features.md` §0)

- `brand.tagline` (es) MUST be exactly `"Estilo americano-japonés"` — never `"Comida japonesa"`.
- `brand.ayceBadge` (es) MUST be exactly `"All You Can Eat"` — never `"Buffet"`.
- Equivalent EN strings: `brand.tagline` (en) = `"American-Japanese style"`; `brand.ayceBadge` (en) = `"All You Can Eat"` (untranslated, brand asset).

### Fallback behavior

- When a key is referenced but missing in the target locale's JSON, the rendered output MUST be the key name (e.g., `nav.unknownKey`) — never a blank string and never a runtime error. This makes gaps visible during development.

## 4. UI Base Component contracts

Every base component under `app/components/ui/`. Each one MUST: declare props via `defineProps<>()` with strict types, have a co-located `*.stories.ts`, stay under 200 lines, and not import from `server/` or any DB client.

## 5. Constraints applied uniformly to every base component

- **Article I**: parameterized via `defineProps<>()`; no per-variant duplicate file.
- **Article II**: TypeScript strict, no `any`, `<script setup lang="ts">`.
- **Article V**: no import from `drizzle-orm`, `@neondatabase/serverless`, or `server/**`.
- **Article VII**: co-located `*.stories.ts` with Default + variants + mobile (360px) + desktop (1200px) viewport stories.
- **Article VIII**: < 200 lines per file; < 30 lines per function; no `console.log`; no TODO; no commented-out code.
- **Article XI**: imports via `@/components`, `@/composables`, `@/types`, `@/utils`, `@/layouts` only.

---

## 6. No persistent state

This feature does not introduce, modify, or migrate any database table. Every "contract" above is a design-time agreement enforced by code review, grep, and Storybook. Future features (009–014) will introduce persistent entities as needed in their own specs.
