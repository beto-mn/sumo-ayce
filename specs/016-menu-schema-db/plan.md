# Implementation Plan: Menu Database Schema & Migration

**Branch**: `chore/011-menu-schema-db` | **Date**: 2026-06-20 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/016-menu-schema-db/spec.md`

---

## Summary

Extend the existing Drizzle ORM schema with three menu tables (`menu_categories`, `dishes`, `sauces`) and three enums. Each table includes a `file_name` (text, nullable) column for resolving local static image assets. A single Drizzle migration is generated and applied directly to the production Neon PostgreSQL database (no Docker). A seed script populates all 13 categories, representative dishes, and 12 sauces. Two query helpers are added for the menu page feature (011).

**Prior art**: `git stash@{0}` (branch `chore/011-menu-schema-drizzle`) contains a reviewer-approved schema draft with the tables and enums but WITHOUT `file_name`. The implementation pops this stash, adds `file_name` to all three tables, then generates and applies the migration.

## Implementation Phases

### Phase 0: Apply stash and extend schema

1. Pop `git stash@{0}` to restore the reviewer-approved schema additions (3 enums + 3 tables, without `file_name`).
2. Add `fileName: text('file_name')` (nullable) to `menuCategories`, `dishes`, and `sauces` in `server/db/schema.ts`.
3. Run `pnpm db:generate` → produces `server/db/migrations/0008_flat_blindfold.sql` + updates `_journal.json`.

### Phase 1: Apply migration

4. Run `pnpm db:migrate` against the production Neon DB (credentials from `.env`).
5. Verify with `drizzle-kit studio` or a direct query that `menu_categories`, `dishes`, and `sauces` tables exist with the correct columns.

### Phase 2: Query helpers + tests (Red → Green)

6. Write failing Vitest tests for `getFeaturedDishes()` and `getFullMenu()` in `tests/db/menu-queries.test.ts`.
7. Implement `server/db/queries/menu.ts` with both helpers.
8. Run `pnpm test` — verify all new tests pass, no regressions.

### Phase 3: Seed

9. Implement `server/db/seed.ts` — 13 categories (with display order, bilingual names), representative menu items, 12 sauces.
10. Run `pnpm db:seed` against the production DB.
11. Run `pnpm db:seed` a second time — verify idempotency (no duplicate rows).

### Phase 4: GitHub Actions migration step

12. Edit `.github/workflows/production.yml` — add a "Run DB migrations" step **before** the Vercel deploy steps that runs `pnpm db:migrate` using the `DATABASE_URL` GitHub Secret.
13. Add `DATABASE_URL` to the GitHub repository secrets (Settings → Secrets and variables → Actions).
14. The step must be ordered so a migration failure cancels the entire workflow before Vercel is touched.
