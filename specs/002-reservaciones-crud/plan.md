# Implementation Plan: Reservaciones — Backend CRUD API

**Branch**: `feat/002-reservaciones-crud` | **Date**: 2026-05-23 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/002-reservaciones-crud/spec.md`

## Summary

Implement a REST API for the reservations system and branch listing within Nuxt 3 server routes (`/server/api/`). This includes: a `GET /api/v1/branches` endpoint for the branch selector, five CRUD endpoints for reservations against Neon PostgreSQL via Drizzle ORM, Zod-based input validation, centralized error handling, a schema migration adding `postal_code` and three indexes to `branches`, and Vitest unit tests covering all endpoints.

## Technical Context

**Language/Version**: TypeScript 5.9 (strict mode) — Nuxt 3 server routes (H3 framework)
**Primary Dependencies**: Drizzle ORM, Zod, `@neondatabase/serverless`, H3 (via Nuxt 3), `@nuxt/test-utils`, Vitest
**Storage**: Neon PostgreSQL (serverless HTTP driver) — `db` utility at `server/utils/db.ts`
**Testing**: Vitest + `@nuxt/test-utils` — unit tests co-located with source (`*.test.ts`)
**Target Platform**: Vercel Serverless (Nuxt 3 server routes)
**Project Type**: Web API — REST endpoints within a Nuxt 3 monorepo
**Performance Goals**: < 500ms p95 response time per SC-001
**Constraints**: Soft-delete required (`deleted_at`), Zod validation at every boundary, centralized error handler (`server/utils/error-handler.ts`), no auth in this phase
**Scale/Scope**: ~10–20 branches, low-to-medium reservation volume (restaurant chain)
