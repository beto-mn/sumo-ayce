# Research: Database Schema & Local Development Environment

**Date**: 2026-05-22 | **Branch**: `feat/001-db-schema-drizzle`

---

## Decision 1: Database Provider

**Decision**: Neon PostgreSQL (free tier, serverless HTTP driver)
**Rationale**: Vercel serverless functions cannot maintain TCP persistent connections. Neon's HTTP driver sends each query over HTTPS — no connection pooling required, no `too many connections` errors. Free tier includes 0.5 GB storage and 190 compute hours/month, sufficient for the project's initial scale. The cPanel MySQL on Hospedando.mx is exclusively for WordPress and must not be used for the backend.
**Alternatives considered**:
- Vercel Postgres (built on Neon internally, but adds Vercel vendor lock-in with no benefit)
- PlanetScale MySQL (MySQL-compatible serverless, but requires changing from PostgreSQL and losing Drizzle's first-class PG support)
- Supabase (PostgreSQL + extras, overkill for this use case, adds unnecessary complexity)
- cPanel MySQL (shared hosting, connection limits, cross-region latency — rejected)

---

## Decision 2: ORM

**Decision**: Drizzle ORM with drizzle-kit for migrations
**Rationale**: Already installed. Lightweight, TypeScript-first, generates plain SQL migrations (auditable, reversible). Works natively with `@neondatabase/serverless` HTTP driver. No Prisma binary needed, which matters in serverless cold starts.
**Alternatives considered**:
- Prisma (heavier, binary dependency, slower cold start, overkill for this schema complexity)
- Raw SQL (no type safety, rejected per Constitution I)
- Kysely (good alternative but Drizzle already chosen and installed)

---

## Decision 3: Primary Keys

**Decision**: UUID v4 (`uuid` type, generated by the database default)
**Rationale**: Serverless functions can create records from multiple instances simultaneously. Sequential integer IDs require a single authoritative counter — in serverless this can cause contention. UUIDs are globally unique by construction. PostgreSQL has native `uuid` type and `gen_random_uuid()` built-in (no extension needed in PG 13+).
**Alternatives considered**:
- Serial integer (sequential, predictable — enumeration attack vector on public endpoints)
- ULID (sortable UUIDs — useful but adds a dependency; defer until needed)
- CUID2 (same tradeoff as ULID)

## Decision 4: Local Development Database

**Decision**: Docker Compose with PostgreSQL 16 official image
**Rationale**: Single command to start (`docker compose up -d`). Persists data via named volume. Uses same PostgreSQL version as Neon production. Credentials loaded from `.env.local`. Developers without Docker can use a free Neon branch instead (documented in quickstart).
**Alternatives considered**:
- PostgreSQL installed directly on host (no isolation, version conflicts across projects)
- Neon branch for dev (valid fallback, documented as alternative in quickstart)
- SQLite locally (different SQL dialect — migration files won't be portable)

---

## Decision 5: Migration Strategy

**Decision**: Incremental SQL migrations via `drizzle-kit generate` + `drizzle-kit migrate`
**Rationale**: Each schema change generates a new numbered SQL file. Migrations are applied in order, never destructive unless explicitly written. The migration history is the source of truth for schema evolution. Files are committed to git.
**Alternatives considered**:
- Push mode (`drizzle-kit push`) — applies schema directly without generating files. Fast for prototyping but no audit trail. Rejected for production use.
- Full reset on each change — destructive, not acceptable once real data exists

---

## Decision 6: Timestamps

**Decision**: `created_at` (not null, default `now()`) and `updated_at` (not null, default `now()`) on all tables. `deleted_at` nullable only on soft-delete entities.
**Rationale**: `updated_at` must be maintained by the application on every UPDATE (Drizzle doesn't auto-update it). This is documented in quickstart. Soft-delete only on: customers, reservations, loyalty transactions. Administrative tables (branches, rewards, staff) use hard delete or `is_active` flag.

---

## Decision 7: Enum Strategy

**Decision**: PostgreSQL native enums via Drizzle `pgEnum`
**Rationale**: Type-safe, stored efficiently in PG, enforced at DB level. Drizzle generates proper migration DDL for enums.
**Alternatives considered**:
- Varchar with check constraint (portable but no type inference)
- Separate lookup tables (overkill for stable, small value sets like status)
