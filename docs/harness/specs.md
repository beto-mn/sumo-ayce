# SDD process with spec-kit

> Spec Driven Development in this project is driven through **spec-kit**.
> Specs are not written by hand — they are generated by invoking the
> `/speckit.*` skills.

## Flow

```
pending → [spec_author wrapping speckit.*] → spec_ready → ⏸ HUMAN → in_progress → [implementer → reviewer] → done
```

## spec-kit skills (what the `spec_author` invokes)

| Skill | What it does | File it produces |
|---|---|---|
| `/speckit.specify <description>` | Generates requirements and acceptance criteria | `specs/<num>-<name>/spec.md` |
| `/speckit.clarify` | Resolves `[NEEDS CLARIFICATION]` markers by asking questions | updates `spec.md` |
| `/speckit.plan <considerations>` | Generates technical plan and Phase -1 gates derived from the constitution | `specs/<num>-<name>/plan.md` |
| `/speckit.tasks` | Decomposes the plan into atomic tasks with `[P]` parallelizable markers | `specs/<num>-<name>/tasks.md` |
| `/speckit.analyze` | Analyzes spec coverage (post-hoc) | report |
| `/speckit.taskstoissues` | Converts tasks to GitHub Issues | remote issues |

## Mandatory order

1. `/speckit.specify` — the "what"
2. `/speckit.clarify` — only if `spec.md` has `[NEEDS CLARIFICATION]`
3. `/speckit.plan` — the "how"
4. `/speckit.tasks` — the executable decomposition

**Never invert the order.** `plan` depends on `spec`. `tasks` depends on `plan`.

## Files per spec

After the 3 skills, a spec folder contains:

```
specs/
└── 00X-feature-name/
    ├── spec.md          # Requirements + acceptance criteria (the "what")
    ├── plan.md          # Architecture + Phase -1 gates (the "how")
    ├── tasks.md         # Atomic tasks with [P] parallelizable
    ├── data-model.md    # Entities and relationships (if applicable)
    ├── research.md      # Technical context, benchmarks, libraries
    ├── quickstart.md    # Quick usage guide
    └── contracts/       # API contracts (OpenAPI / Markdown)
        └── api.md
```

Numbers are sequential (`001-`, `002-`, ...) and spec-kit assigns them.
**Never create spec folders by hand** — always via `/speckit.specify`.

## Phase -1 Gates

`plan.md` includes gates automatically derived from the constitution. Example:

```markdown
### Phase -1: Pre-Implementation Gates

#### Code Organization (Article I — NON-NEGOTIABLE)
- [ ] Feature lives entirely in `app/features/<name>/` and `server/api/v1/<name>/`?
- [ ] No cross-feature imports?
- [ ] Components parameterized via props (no duplicate variant files)?

#### Constitutional Compliance (Article II — TypeScript strict)
- [ ] No `any` types?
- [ ] Vue components in `<script setup lang="ts">`?

#### Test Gate (Article IV)
- [ ] Tests written BEFORE production code?
- [ ] Tests failing (Red phase) before implementing?

#### UI Documentation (Article VII)
- [ ] Every new UI component has a story?
- [ ] The story covers default + variants + breakpoints?
```

The `reviewer` verifies all Phase -1 gates are `[x]` before approving.

## EARS notation (reminder)

Although spec-kit writes the requirements, it helps to recognize EARS notation:

```
[WHILE <pre-condition>] WHEN <trigger> THE SYSTEM SHALL <response>
```

Examples:
- `WHEN the user submits the reservation form THE SYSTEM SHALL validate party_size between 1 and 20`
- `WHILE the session is active WHEN the auth token expires THE SYSTEM SHALL refresh it automatically`
- `IF the branch has no manager phone WHEN a reservation is confirmed THE SYSTEM SHALL log a warning and skip the notification`

## Human approval gate

After `/speckit.tasks`, the `spec_author` changes the status to `spec_ready`
and stops. **The human must read all of `specs/<num>-<name>/` and explicitly
approve** ("approved") before the `leader` changes to `in_progress`.

If the human requests changes, the `spec_author` regenerates only what's needed
(typically `spec.md` or `plan.md` or `tasks.md` individually, not all 3).

## Constitution as source of truth

The constitution (`.specify/memory/constitution.md`) **replaces**
`docs/harness/architecture.md` and `docs/harness/conventions.md`. If you want
to change project principles, invoke `/speckit.constitution` — don't edit the
stubs in `docs/harness/`.

## SDD anti-patterns

- ❌ Creating `specs/<num>-name/` by hand without going through `/speckit.specify`.
- ❌ Implementing code before having an approved `spec_ready`.
- ❌ Skipping `/speckit.clarify` "because the clarifications are obvious".
- ❌ Editing `tasks.md` by hand after implementing — regenerate with `/speckit.tasks` if scope changes.
- ❌ Marking `done` with Phase -1 gates still `[ ]`.
