The tag-trigger model is REJECTED.
2. **Release-note body template**: replace the simple "description + optional append" body with a structured template generated by a bash step into `release_body.md` (NOT `/tmp/`), passed to `softprops/action-gh-release@v2` via `body_path:`. Title `🚀 v<X.Y.Z> - <Patch|Minor|Major> Release (Production)`; body has `## {EMOJI} {Type} Release` header (semantic emoji: 🐛/✨/🚨), Version/Type/Environment block, `## 📝 Changes` from `release_description`, `## 📋 Commit History` auto-generated via `git log "${PREV_TAG}..HEAD^"` (with first-release fallback to last 50 commits + commits-view URL), Deployment block, optional `## 📌 Additional Notes` block before bot signature.
3. **Deployment block fields**: `Environment: prd` and `Service: sumo-ayce` ONLY. NO Region line — Vercel handles edge regions automatically; showing one is misleading.

## Inconsistencies surfaced during the edit

- The original C4.9 contract specified `body_path: /tmp/release_body.md`. The new design writes to `release_body.md` in the repo working directory. The new contract C4.9 says `release_body.md` explicitly; the old T407 task (which referenced `/tmp/`) is reopened with the invalidation note pointing to T901 for the new implementation.
- The original C2.7/C3.7 used `${{ github.ref }}` for the production checkout (because the workflow ran on tag push and `github.ref` resolved to the tag). The new C3.7 uses `${{ inputs.tag }}` because `workflow_call`/`workflow_dispatch` don't have a meaningful `github.ref` for the deploy target — the tag is now an explicit input.
- The original C3.3 concurrency group used `github.ref`; the new C3.3 uses `inputs.tag`. This means a `workflow_dispatch` redeploy for an existing tag while another redeploy for the SAME tag is mid-flight will cancel the in-flight one (correct behaviour: "latest invocation for this tag wins").
